El proceso, de punta a punta

El proceso, en 3 fases

1

De scope a spec

Análisis → ideación → especificación.

Pasamos de una idea vaga a un documento que el agent pueda ejecutar.

2

Crítica y refinamiento

Reflection loop, multi-round.

El agent ataca la spec hasta que no encuentra más huecos.

3

Planning → implementation

Plan revisado → código.

Si las fases 1 y 2 están bien, ésta se simplifica.

Hoy nos enfocamos en fases 1 y 2. Es donde se gana o se pierde el proyecto.
Fase 1 · De scope a spec 05 / 21
Paso 1 · Análisis

Análisis: las 3 preguntas

Antes de comenzar a implementar con el coding agent, asegúrate de poder responder estas 3 preguntas.

01

¿Qué quiero lograr?

Outcomes concretos, resultados esperados, métricas verificables.

02

¿Qué no quiero?

Límites, out of scope, constraints. Tan importante como el "qué sí".

03

¿Cómo sé que está "done"?

Criterios de verificación, tests, success criteria medibles.

Regla de oro · si no puedes responder las tres con claridad, no estás listo para el agent. Piensa primero, ejecuta después.

Técnica 09 / 21
Técnica avanzada de crítica

Role-based critique

Diferentes roles = diferentes perspectivas. Un mismo agent encuentra issues distintos según el sombrero que le pongas.

Security engineer
“Revisa esta spec como un security engineer. ¿Qué vulnerabilidades ves?”

Inputs sin validar, falta de rate limiting, credenciales hardcoded.

UX designer
“Revisa esta spec como un UX designer. ¿El flujo es intuitivo? ¿Qué frustraciones tendrá el usuario?”

5 clicks para tareas simples, sin feedback visual, controls escondidos.

Performance engineer
“Revisa esta spec como un performance engineer. ¿Qué podría escalar mal?”

Carga full sin paginación, N+1 queries, sin caching.

Tip de model settings · temperature alta (0.8–1.0) para crítica creativa, baja (0.2–0.4) para implementación.
Técnica 10 / 21
Técnica · pensar en voz alta

Chain of thought

Haz que el agent razone explícitamente antes de responder. Reasoning paso a paso = respuestas más profundas.

✗ Prompt débil
“¿Esta spec está completa?”

El agent contesta superficial: “Sí” / “Falta X”. No explora.

✓ Prompt fuerte
“Antes de responder, piensa paso a paso:
  1. ¿Qué componentes necesita el sistema?
  2. ¿Cada componente está especificado?
  3. ¿Las interacciones están claras?
  4. ¿Los edge cases están cubiertos?

Ahora sí, responde si está completa.”

Razonamiento explícito antes de la conclusión.

Por qué funciona · los LLMs son mejores cuando razonan antes de responder. Si no se lo pedís, eligen la ruta más corta.
Técnica 11 / 21
Técnica · reglas explícitas

Three-tier boundaries

Tres categorías de reglas que el agent respeta sin tener que adivinar.

✓ Always

Autonomía total. Cosas seguras que debe hacer siempre.

Use Alembic para schema changes.
Trigger: database, schema, table, column.

Reduce fricción en lo predecible.

⚠ Ask first

Alto impacto. Pide confirmación antes de actuar.

Dropping columns (data loss).
Trigger: DROP, delete column, remove field.

Previene cambios grandes sin que lo sepas.

✕ Never

Hard stop. Prohibido absolutamente.

Raw SQL en application code.
Trigger: cursor.execute, raw query, SQL string.

Safety crítico, sin excepciones.

Tip avanzado · trigger prompts · enumerá las palabras clave que disparan cada regla. El agent las detecta proactivamente, no espera a meter la pata.
Model settings 12 / 21
Lo que cambia entre fase y fase

Model settings que importan

Temperature es el setting más importante. Controla qué tan creativo vs determinista es el agent.

0.0
Implementation 0.2 – 0.4
Ideación & Critique 0.8 – 1.0
1.0
Ideación & critique

Temperature alta

Querés exploración, edge cases, perspectivas distintas. Mismo prompt × 3 = 3 respuestas distintas.

  • Top-p 0.9 – 0.95
  • Presence penalty 0.1 – 0.3
  • Max tokens 4000 – 8000
Implementation

Temperature baja

Querés determinismo, código consistente. Que siga la spec al pie de la letra, sin improvisar.

  • Top-p 0.9
  • Presence penalty 0
  • Max tokens según task
Opus 4.7 · abril 2026 · más conservador por default. Para critique agrega: “if you can think of an ambitious analysis… prefer that over listing small individual gaps.” Sin esto, da análisis muy atómicos.

Context building

Para proyectos largos no alcanza con una spec por feature. Necesitás documentos que mantengan alignment a través de sesiones.

📜project_constitution.md

Project constitution

  • Mission & valores
  • Tech philosophy (ej: simplicidad sobre features)
  • Decisiones arquitecturales fundamentales
  • Roadmap 6–12 meses
🎯strategy/expense.md

Strategy docs

  • Por qué construir esta feature ahora
  • Qué problema del usuario resuelve
  • Cómo se conecta con el resto del sistema
  • Contexto de negocio
📐guides/react.md

Style guides & patterns

  • Convenciones de naming
  • Patterns establecidos (auth, errors)
  • Ejemplos concretos del repo (few-shot)
  • Cómo hacemos las cosas aquí
Reduce context drift · sin estos docs, cada sesión empieza de cero. Con ellos, hay continuidad y el agent produce código alineado con la cultura del proyecto.
Harness engineering 16 / 21
Lo que aprendés se convierte en guía

Harness engineering: el learning loop

El agent lee AGENTS.md / CLAUDE.md en cada sesión. Lo que documentes ahí compone el "harness" que mantiene al agent on-track.

  • · Decisiones tomadas — y por qué
  • · Patterns que funcionaron
  • · Anti-patterns que no funcionaron
  • · Edge cases descubiertos
  • · Constraints específicos del proyecto

El agent no repite mistakes y sigue patterns que ya sabés que funcionan.

AGENTS.md 
## Authentication
- JWT expiry: 1 hour (security req)
- Refresh token: 7 days
- Always use httpOnly cookies for tokens
- Never store tokens in localStorage (XSS risk)

## Tech stack decision
- Chose Firebase over custom backend
- Rationale: speed to market + mobile
- Implication: Zustand for state (simpler)
"El agent no necesita más instrucciones. Necesita un mundo donde hacer lo correcto es obvio y lo incorrecto es difícil." — paráfrasis del approach de OpenAI Harness Engineering

Red flags

Seis anti-patterns frecuentes. Evítalos y la experiencia con coding agents mejora drásticamente.

✕

“Hazme algo genial”

Demasiado vague. El agent adivina y rara vez acierta.

✕

Specs de 50 páginas en un solo prompt

Context overload. El agent se pierde y prioriza mal.

✕

Asumir que tests = correctness

Los tests pueden ser incompletos. Revisa siempre.

✕

Skipear el review del plan

Fixing errores en código es 10–100× más caro.

✕

Mismo temperature para todo

Critique necesita alta. Implementation, baja.

✕

No versionar la spec

Pierdes track de decisiones y por qué cambiaron.

Slide 01

[1 minuto]

Hoy vamos a hablar de cómo trabajar con coding agents de manera profesional.

Andrej Karpathy - el tipo que inventó el término "vibe coding", ex-director de IA en Tesla y fundador de OpenAI - acaba de advertir en 2026 que hay un gap creciente entre usuarios de IA.

De un lado: Casual users que probaron ChatGPT gratis hace un año y piensan que la IA es puro hype, llena de errores y poco confiable.

Del otro lado: Power users que pagan por Claude Code, usan workflows estructurados, y tienen 3 a 5 veces más productividad.

La idea es este espacio de charla es entender de que forma podemos cruzar ese gap para ganar esa productividad.

La idea de base es simple: La calidad del código que produce el agent es directamente proporcional a la calidad de tu especificación.

Si tu spec (especificacion) es vaga, el código será inconsistente. Si tu spec es clara, el código será predecible y mantenible, al igual que la salida de lo que esperesl

No vamos a hablar de "pedirle cosas a Claude y esperar que funcione". Vamos a hablar de cómo pensar, planificar, y especificar ANTES de que el agent escriba una sola línea de código.

Slide 02

[2 minutos]

Levanten la mano los que alguna vez le pidieron algo a un coding agent y el resultado no fue lo que esperaban. [Pausa para manos]

Esto es súper común. Y no es culpa del agent. Es que le dimos instrucciones vagas.

Imaginen que van a un restaurante y dicen "haceme algo rico". El chef va a adivinar. Quizás te trae pizza, quizás sushi, quizás un bife. Podría ser bueno, podría no ser lo que querías.

Ahora imaginen que dicen: "Quiero spaghetti carbonara: pasta al dente, huevos, pancetta crocante, pecorino, pimienta negra. Sin crema." El chef sabe EXACTAMENTE qué hacer.

Lo mismo pasa con coding agents. Instrucciones vagas = resultados impredecibles.

La analogía de la receta de cocina es perfecta para entender spec-driven development. Y es exactamente lo que vamos a aprender hoy: cómo escribir "recetas" claras para que los agents produzcan exactamente lo que necesitamos.

Slide 03

[1.5 minutos]

Esto no es teoría. Hay data real de 2026 que muestra el impacto.

Teams que escriben specs antes de usar agents tienen 67% menos rollbacks segun datos de industria - es decir, 67% menos veces que tienen que tirar código y empezar de nuevo.

Teams que usan spec-driven workflows son 3 a 5 veces más productivos que teams que usan AI como simple autocomplete.

Y acá viene lo interesante: OpenAI publicó en febrero 2026 cómo su equipo de Codex - solo 3 engineers - shipped 1 millón de líneas de código usando exactamente el approach que vamos a ver hoy. La clave fue usar archivos AGENTS.md para guiar el comportamiento del agent. Zero líneas escritas a mano.

Y LangChain hizo un benchmark fascinante: tomaron el mismo modelo dos veces, cambiaron SOLO el harness - el contexto y las guías - y el performance saltó de 52.8% a 66.5%. Un 13.7% de improvement solo cambiando el contexto.

El término de industria para esto es "Harness Engineering". Y el consenso es claro: "The harness design represents the actual competitive advantage, not raw model capability."

El mensaje es claro: invertir tiempo en specs y harness te ahorra MUCHÍSIMO tiempo después.

Slide 04

[1 minuto]

El proceso tiene 3 fases principales, pero hoy nos vamos a enfocar en las dos primeras.

Fase 1 es pasar de una idea vaga a una especificación clara. Esto tiene tres pasos: Análisis (entender el problema), Ideación (explorar soluciones), y Especificación (bajar todo a un documento claro).

Fase 2 es crítica y refinamiento. Acá usamos al mismo y otro agente para criticar nuestra especificación, buscar huecos, identificar puntos ciegos. Hacemos múltiples rondas hasta estar satisfechos.

Fase 3 es planning e implementation, pero si las fases 1 y 2 están bien hechas, esta fase se simplifica, creo que la importancia que tiene esencialmente en planificacion es entender como incrementalmente queremos ir revisando y creando la solucion.

Slide 05

[2 minutos]

El primer paso es Análisis. Y acá tenemos un framework súper simple: Las 3 Preguntas.

Pregunta 1: ¿Qué quiero lograr?
No "quiero una app". Sino: "Quiero que los usuarios puedan registrar gastos en menos de 10 segundos, categorizar automáticamente, y ver un resumen mensual."

Eso es un outcome concreto. Medible. Verificable.

Pregunta 2: ¿Qué NO quiero?
Esto es igual de importante. "NO quiero que la app maneje pagos reales, solo tracking. NO quiero integraciones con bancos por ahora. NO quiero múltiples usuarios, solo uso personal."

Los límites son críticos. Le dicen al agent qué NO hacer.

Pregunta 3: ¿Cómo sé que está done?
"Está done cuando puedo agregar un gasto, verlo en la lista, y el total del mes se actualiza correctamente. Está done cuando funciona en mobile. Está done cuando carga en menos de 2 segundos."

Regla de oro: Si no podés responder estas 3 preguntas con claridad, no estás listo para pedirle código al agent. Primero pensá, después ejecutá.

Slide 06

[2 minutos]

El paso 2 es Ideación. Acá es donde mucha gente se apresura. Dicen "ok, quiero una app web" y arrancan.

Mejor approach: Usá al agent como thinking partner ANTES de decidir.

Por ejemplo, este prompt: "Quiero trackear gastos personales. Explorá 3 approaches diferentes: una app web simple, un bot de Telegram, o Google Sheets con Apps Script. Para cada uno, dame pros, cons, y estimación de complejidad."

El agent va a pensar en trade-offs que quizás no consideraste:
- App web = más flexible, pero necesitás hosting
- Bot de Telegram = super rápido de usar, pero limitado en UI
- Google Sheets = cero setup, pero menos control

Este es el momento de explorar. No de committearse a una solución. Dejá que el agent te muestre opciones.

Y acá viene un tip importante: Usá role prompting. Podés decir "Actuá como un arquitecto de software con 10 años de experiencia. Qué approach recomendarías y por qué?"

Diferentes roles dan diferentes perspectivas. Probá "actuá como un designer de UX", "actuá como un performance engineer". Cada uno va a resaltar cosas distintas.

Slide 07

[2 minutos]

Ahora sí, paso 3: Especificación. Bajar todo a un documento claro.

GitHub analizó más de 2,500 proyectos con coding agents y encontró que las specs exitosas tienen 7 elementos:

1. Outcomes: Qué debe lograr el sistema
2. Scope Boundaries: Qué está in scope, qué out of scope
3. Constraints: Tech stack que tenés que usar, requirements de performance, deadlines
4. Prior Decisions: Las decisiones arquitecturales ya tomadas y POR QUÉ
5. Task Breakdown: Fases, milestones, dependencies
6. Edge Cases: Casos extremos que hay que considerar
7. Verification Criteria: Tabla que mapea cada requirement a un test específico

El séptimo elemento es posiblemente el más importante: Verification Criteria. Esto es lo que transforma tu spec de un documento en algo ejecutable.

[MOSTRAR EN SLIDE: Ejemplo de tabla]
```
| Requirement | Test Type | Test Case | Success Criteria |
|-------------|-----------|-----------|------------------|
| Valid registration | Unit | test_register_valid | User created, returns user_id |
| Duplicate email | Unit | test_duplicate_email | Returns 409, "Email exists" |
| Email format | Unit | test_email_format | Rejects invalid formats |
```

Para cada requirement, defines: qué tipo de test necesitás, qué caso específico vas a probar, y qué significa éxito. Esto conecta directamente con la fase 3 donde generás los tests ANTES de implementar.

Formato recomendado: Markdown simple. Fácil de leer, fácil de editar, funciona en cualquier editor.

Versionado: Git. Tratá tu spec como código. Cada cambio, un commit. Esto te permite volver atrás si una decisión no funcionó.

Tip importante: Incluí ejemplos concretos en la spec. No digas "la app debe ser rápida". Decí "la app debe cargar la lista de gastos en menos de 2 segundos con 1000 entries".

Ejemplos concretos le dan al agent referencias claras. Esto se llama few-shot prompting: le mostrás ejemplos de lo que querés, no solo le describís con palabras.

Slide 08

[3 minutos]

Ahora viene la parte más importante: Crítica Multi-Round.

Acá es donde usás al agent para mejorar tu propio pensamiento ANTES de pedirle que escriba código.

El loop es simple:

Round 1 - Gaps: "Revisá esta spec. ¿Qué estoy omitiendo? ¿Qué requirements faltan?"

El agent va a encontrar cosas. "No especificaste qué pasa si el usuario ingresa un gasto negativo. No mencionaste qué categorías de gastos vas a tener. No definiste el formato de fecha."

Actualizás la spec.

Round 2 - Edge Cases: "¿Qué podría salir mal? ¿Qué edge cases no estoy considerando?"

El agent piensa en escenarios: "Qué pasa si el usuario ingresa 50 gastos el mismo día? Qué pasa si cambian de timezone? Qué pasa si borran un gasto por error?"

Actualizás la spec otra vez.

Round 3 - Blind Spots: "¿Qué no estoy considerando? ¿Qué suposiciones estoy haciendo que podrían ser incorrectas?"

El agent cuestiona tus assumptions: "Asumiste que todos los gastos son en una sola moneda. Asumiste que el usuario siempre tiene internet. Asumiste que nadie más va a usar la app."

Y podés seguir. No hay número mágico de rounds. Parás cuando estás confiado de que la spec es sólida.

Dato clave: Research muestra que este proceso de reflection mejora la calidad del código final de 80% a 91%. Es decir, casi 10 puntos porcentuales de mejora solo por hacer crítica antes de codear.

Pro tip para los más avanzados: Pueden hacer rondas de critique con diferentes modelos - Claude, GPT-4, Gemini. Cada uno tiene fortalezas distintas, y esto reduce blind spots. Es como tener diferentes expertos revisando tu spec.

Slide 09

[2 minutos]

Una técnica super poderosa es Role-Based Critique.

En vez de solo decir "revisá la spec", le das al agent un rol específico.

"Revisá esta spec como un security engineer. ¿Qué vulnerabilidades o security gaps ves?"

El agent va a pensar desde esa perspectiva: "No hay validación de inputs. No hay rate limiting. Las credenciales están hardcoded."

"Ahora revisá como un UX designer. ¿Es intuitivo el flujo? ¿Qué frustraciones tendrá el usuario?"

Perspectiva completamente diferente: "El usuario tiene que hacer 5 clicks para un gasto simple. No hay feedback visual cuando se guarda. El botón de categoría está escondido."

"Ahora como performance engineer. ¿Qué podría causar problemas de escalabilidad?"

Otra vez, diferentes issues: "Estás cargando todos los gastos del año de una vez. No hay paginación. Cada gasto hace una query separada a la DB."

Cada rol encuentra problemas distintos. Es como tener un equipo completo revisando tu spec, pero todo desde el mismo agent.

Tip de model settings: Para role-based critique, usá temperature más alta (~0.8-1.0). Querés que el agent sea creativo y piense en escenarios diversos. Para implementation después, bajás temperature (~0.2-0.4) para que sea más deterministic.

Slide 10

[2 minutos]

Otra técnica crítica: Chain of Thought.

La idea es hacer que el agent piense en voz alta antes de darte una respuesta.

Prompt débil: "¿Esta spec está completa?"

El agent te va a decir "Sí, parece completa" o "Falta mencionar X". Es superficial.

Prompt fuerte: "Antes de responder si la spec está completa, pensá paso a paso: Primero, ¿qué componentes necesita el sistema? Segundo, ¿cada componente está especificado claramente? Tercero, ¿las interacciones entre componentes están documentadas? Cuarto, ¿los edge cases están cubiertos? Ahora sí, respondé si está completa."

Cuando el agent tiene que pensar paso a paso, encuentra mucho más. Es como si le estuvieras diciendo "no me des la respuesta rápida, pensá profundo".

Por qué funciona: Los LLMs son mejores cuando razonan explícitamente. Si solo les preguntás "¿está bien?", van por el camino más corto. Si les pedís que razonen paso a paso, activan mejor su "pensamiento".

Aplicalo en critique: "Antes de decirme qué falta, primero identificá todos los user flows. Después, para cada flow, chequeá si está completo. Después chequeá dependencies. Después chequeá error handling."

Vas a tener respuestas mucho más profundas.

Slide 11

[2 minutos]

Una técnica súper práctica es el Three-Tier Boundary System.

Le das al agent tres categorías de reglas claras:

✅ ALWAYS - Cosas que debe hacer siempre sin preguntar:
- "Siempre corré todos los tests después de hacer cambios"
- "Siempre seguí las convenciones de naming del proyecto"
- "Siempre usá los patterns que ya existen en el código"

Esto le da autonomía en cosas que son safe.

⚠️ ASK FIRST - Cosas de alto impacto que requieren aprobación:
- "Pedí aprobación antes de cambiar el schema de la base de datos"
- "Consultá antes de agregar una nueva librería externa"
- "Avisá si necesitás modificar la arquitectura"

Esto previene que el agent haga cambios grandes sin que vos sepas.

🚫 NEVER - Hard stops, prohibido absolutamente:
- "NUNCA commiteés passwords o secrets"
- "NUNCA edites archivos en node_modules/ o vendor/"
- "NUNCA cambies configuraciones de producción"

Esto es safety crítico.

Por qué funciona: Reduce la "creatividad" del agent en lugares donde no la querés. El agent no tiene que adivinar "¿puedo hacer esto?". Tiene reglas claras.

Y acá va un tip avanzado: Agregá "trigger prompts" a cada boundary. Le estás diciendo al agent: "cuando veas estas palabras clave en el contexto, aplicá esta regla automáticamente".

Por ejemplo, en el NEVER: "Trigger: When you see: cursor.execute, raw query, SQL string" - Cuando el agent ve esas palabras, sabe que no debe usar raw SQL.

Esto hace que las boundaries sean proactivas - el agent no espera a meter la pata, detecta el pattern y aplica la regla. Es como poner speed bumps en los lugares correctos.

Incluilo en tu spec: Tené una sección que se llame "Boundaries" con estas tres categorías y sus triggers. El agent las va a respetar.

Slide 12

[2 minutos]

Hablemos un segundo de model settings porque esto hace diferencia.

Temperature es el setting más importante. Controla qué tan "creativo" vs "determinista" es el agent.

Para Ideación y Critique (Fases 1 y 2): Usá temperature alta (0.8 a 1.0)

¿Por qué? Porque querés que el agent explore opciones diversas, piense en edge cases creativos, te dé perspectivas que no consideraste.

Temperature alta = más variedad en las respuestas. Corrés el mismo prompt 3 veces, vas a tener 3 respuestas distintas. Eso es BUENO en critique porque cada respuesta puede encontrar issues diferentes.

Para Implementation (Fase 3): Usá temperature baja (0.2 a 0.4)

¿Por qué? Porque querés código consistente y predecible. No querés que el agent "se ponga creativo" con la arquitectura. Querés que siga la spec al pie de la letra.

Temperature baja = más determinismo. Corrés el mismo prompt 3 veces, vas a tener respuestas muy similares.

Otros settings útiles:
- Max tokens: 4000-8000 para specs largas (que el agent pueda generar respuestas completas)
- Top-p: 0.9-0.95 (balanceado entre diversidad y coherencia)
- Presence penalty: 0.1-0.3 (evita que el agent se repita)

Tip práctico en Claude Code: Cuando estés en fase de critique, literalmente decí en el prompt "sé creativo, pensá en escenarios diversos". Cuando estés en implementation, decí "seguí la spec exactamente, no improvises".

Actualización Abril 2026 - Opus 4.7: El nuevo modelo es más conservador por default. Si usás Opus 4.7 para critique, agregá explícitamente: "If you can think of an ambitious analysis that covers multiple areas or reveals fundamental assumptions, prefer that over listing small individual gaps." Sin esto, Opus 4.7 va a dar análisis muy atómicos y conservadores. En cambio, para verification tasks (tests, PR review), Opus 4.7 es excelente - tiene 98.5% accuracy en precise UI tasks vs 54.5% del anterior.

Slide 13

[2 minutos]

Ok, ya tenés tu spec refinada después de múltiples rounds de critique. Estás confiado. Ahora qué?

No le digas al agent "empezá a codear" todavía.

Primero, pedile un plan.

En Claude Code específicamente, tenés el comando /plan.

Le das la spec y le decís "/plan - basandote en el spec x cómo implementarías esto?"

El agent te va a dar:
- Breakdown de tasks (qué hacer primero, segundo, tercero)
- Files que va a crear o modificar
- Dependencies entre tasks
- Risks que identifica

Y VOS REVISÁS EL PLAN ANTES DE QUE ESCRIBA CÓDIGO.

Esto es critical. Si el agent está por elegir la base de datos equivocada, o está por usar una arquitectura que no escala, lo catcheás en el plan.

Cambiar un plan = 30 segundos. Cambiar código = horas de refactoring.

Data: Catching errores en la fase de planning es 10 a 100 veces más barato que fixing después.

Best practice: Siempre /plan antes de medium o large tasks. Para tasks muy chicos (agregar un botón), podés skipear. Pero para features completas, SIEMPRE plan first.

Slide 14

[3 minutos]

Veamos un ejemplo end-to-end conceptual: una app de expense tracking.

Fase 1 - Scope to Spec:

Empezamos con las 3 preguntas:
- Qué quiero: Users puedan registrar gastos, categorizarlos, ver resumen mensual
- Qué NO quiero: Sin pagos reales, sin múltiples usuarios, sin integraciones bancarias
- Cómo sé que funciona: Puedo agregar gasto en <10 seg, categorización funciona, totales calculan correctamente

Ideación: Exploramos 3 opciones con el agent (web app, bot de Telegram, Google Sheets). Decidimos PWA (Progressive Web App) porque funciona en mobile y desktop, y puede trabajar offline.

Spec: Escribimos los 6 elementos en spec.md. Outcomes, boundaries, constraints (tech stack React + Firebase), prior decisions (por qué PWA), task breakdown, verification criteria.

Fase 2 - Critique Multi-Round:

Round 1: "¿Qué estoy omitiendo?"
- Agent encuentra: no especificaste validación de inputs (gastos negativos?), no mencionaste cómo manejar diferentes monedas
- Actualizamos spec

Round 2: "¿Qué podría salir mal?"
- Agent: edge cases de timezone (usuario viaja a otro país?), múltiples gastos el mismo día, qué pasa si se borra categoría que tiene gastos
- Actualizamos spec

Round 3: "¿Qué no consideré?"
- Agent: performance con 10K+ gastos, qué pasa sin internet (offline support?), backup de datos
- Actualizamos spec

Fase 3 - Planning:

Le damos la spec final al agent con /plan.

Agent propone: Primero DB schema (categorías, gastos, users). Después API endpoints (create expense, list expenses, get summary). Después UI components.

Revisamos el plan. Todo tiene sentido. Aprobamos.

Fase 4 - Implementation:

Agent ejecuta. Va paso a paso. Después de cada milestone, corre tests. Verifica que cumple la spec.

Nosotros hacemos checkpoints. "¿Los tests pasan? ¿Cumple los acceptance criteria de la spec?"

Al final, tenemos una app que funciona exactamente como la especificamos.

Tiempo total: Quizás 2-3 horas incluyendo todas las fases. Si hubiéramos ido directo a "Claude haceme una app", probablemente 2 días de trial and error.

Slide 15

[3 minutos]

Ahora hablemos del siguiente nivel: Context Building.

Hasta ahora hablamos de specs para features individuales. Pero cuando trabajás en proyectos más grandes, necesitás documentos de contexto más amplio.

PROJECT CONSTITUTION:

Es un documento de alto nivel que define:
- Mission: Por qué existe este proyecto
- Tech philosophy: Principios que guían decisiones (ej: "Preferimos simplicidad sobre features", "Performance es crítico")
- Decisiones arquitecturales fundamentales: Por qué usamos X framework, por qué esta estructura
- Roadmap: Hacia dónde va el proyecto en los próximos 6-12 meses

¿Por qué importa? Porque el agent puede mantener alignment filosófico a través de múltiples features. Si el constitution dice "siempre optimizamos para mobile first", el agent va a tener eso en mente en cada feature.

STRATEGY DOCS:

Documentos que explican:
- Por qué estamos construyendo esta feature ahora (timing, business value)
- Qué problemas del usuario resuelve
- Cómo se conecta con otras partes del sistema

Esto le da al agent contexto de negocio. No solo "construí esto", sino "construí esto porque resuelve X problema para Y usuarios".

STYLE GUIDES & PATTERNS:

Documentación de:
- Convenciones de código (naming, estructura de files)
- Patterns arquitecturales establecidos (cómo hacemos auth, cómo manejamos errors)
- Ejemplos concretos de código bien hecho del proyecto

Esto es few-shot prompting a escala de proyecto. Le mostrás al agent "así es cómo hacemos las cosas acá".

Cómo usar estos docs:

En tu prompt inicial para una feature, decís:

"Antes de empezar, leé:
- project_constitution.md (nuestra filosofía)
- strategy/expense_tracking.md (por qué esta feature)
- guides/react_patterns.md (cómo escribimos React)

Ahora sí, implementá según spec.md"

El agent va a tener contexto completo y va a producir código que no solo funciona, sino que está alineado con la cultura del proyecto.

Beneficio clave: Esto reduce "context drift". Sin estos docs, cada sesión con el agent es como empezar de cero. Con estos docs, hay continuidad.

Slide 16

[3 minutos]

Acá viene un punto súper importante que diferencia proyectos one-off de proyectos sostenibles: el Learning Loop.

La idea es simple pero poderosa: Lo que aprendés en cada fase se convierte en guidelines para el agent.

Esto es lo que la industria llama "Harness Engineering". Y no es invento mío - es exactamente lo que OpenAI publicó en febrero 2026.

Cómo funciona:

Después de completar una feature, documentás learnings en archivos específicos: AGENTS.md o CLAUDE.md (depende de la herramienta).

¿Qué documentás?

Decisiones tomadas y por qué:
"Decidimos usar local storage en vez de cookies para preferencias de usuario porque necesitamos más de 4KB de espacio."

Patterns que funcionaron:
"El pattern de validation en el cliente antes de enviar al backend redujo 80% de errores. Usarlo siempre en forms."

Anti-patterns que NO funcionaron:
"NO usar inline styles en React components. Rompe el theming. Siempre usar styled-components o CSS modules."

Edge cases descubiertos:
"Si el usuario ingresa fecha futura, mostrar warning. Caso descubierto en testing."

Constraints específicos del proyecto:
"API tiene rate limit de 100 req/min. Implementar debouncing en search."

Por qué importa:

En la próxima sesión con el agent, en tu prompt inicial decís:

"Leé AGENTS.md antes de empezar. Seguí los patterns documentados."

El agent va a tener context de learnings previos. No va a repetir mistakes. Va a seguir patterns que ya sabés que funcionan.

OpenAI lo dice así: "El agent no necesita más instrucciones. Necesita un mundo donde hacer lo correcto es obvio y lo incorrecto es difícil."

Eso es exactamente lo que estás construyendo con AGENTS.md: guías de rieles que mantienen al agent on-track.

Ejemplo real:

Session 1: Construís feature de authentication. Descubrís que JWT tokens deben tener 1-hour expiry para security.

Documentás en AGENTS.md:
```
## Authentication
- JWT expiry: 1 hour (security requirement)
- Refresh token: 7 days
- Always use httpOnly cookies for tokens
- Never store tokens in localStorage (XSS risk)
```

Session 2: Construís feature de user profiles. Agent lee AGENTS.md. Automáticamente implementa auth correctamente sin que vos tengas que re-explicar todo.

El agent aprende de lo que descubriste antes.

Slide 17

[2.5 minutos]

Otra técnica crítica que hace diferencia: Proceso Iterativo 1-a-1 en las fases de Análisis e Ideación.

El anti-pattern común:

Dump toda la información junta: "Quiero una app que haga X, Y, Z, debe tener features A, B, C, considerando constraints D, E, F..."

El agent recibe TODO de golpe y tiene que adivinar qué priorizar.

El better approach: 1-a-1 Interactive:

Vas pregunta por pregunta, dando feedback incremental.

Ejemplo de flow:

Tu pregunta inicial: "Para una app de expense tracking, qué tech stack recomendás?"

Agent responde: "Opciones: React SPA + Firebase, Next.js + Supabase, o Vue + custom backend. Depende de tus priorities..."

Vos das contexto: "Priorizo speed to market y no quiero manejar backend. ¿Qué cambia?"

Agent refina: "Entonces React + Firebase o Next.js + Supabase. Ambos son managed backends. Firebase tiene mejor ecosystem para mobile si pensás expandir..."

Vos das más contexto: "Mobile es importante. Vamos con Firebase."

Agent continúa: "Perfecto. Para state management con Firebase, recomiendo Zustand sobre Redux porque..."

Vos das feedback: "Zustand me gusta por simplicidad. Confirmado."

¿Ven el pattern?

Cada respuesta del agent incorpora el contexto de la anterior. Cada decisión tuya informa la siguiente pregunta del agent.

Al final, tienen una spec mucho más aligned porque el agent entendió no solo QUÉ querés, sino POR QUÉ tomaste cada decisión.

Beneficio crítico:

Cuando después el agent genera el plan o escribe código, sabe el rationale detrás de cada choice. No solo "usa Firebase" sino "usa Firebase PORQUE priorizamos speed to market y necesitamos mobile support".

Ese rationale lo hace tomar mejores micro-decisions en implementation.

Tip práctico:

En Análisis/Ideación, no tengas prisa. Dedicá 10-15 minutos a este back-and-forth. Vas a ahorrar HORAS después en refactoring.

Y después de la sesión, documentás este rationale en AGENTS.md:

```markdown
## Tech Stack Decision
- Chose Firebase over custom backend
- Rationale: Speed to market + mobile support priority
- Implication: Use Zustand for state (simpler than Redux)
```

Próxima feature, el agent ya sabe el philosophy y continúa consistent.

Slide 18

[2 minutos]

Antes de cerrar, hablemos de Red Flags - cosas que NO hacer.

Red Flag 1: Prompts vagos tipo "haceme algo copado". Ya lo discutimos. El agent no puede leer tu mente.

Red Flag 2: Dumping una spec de 50 páginas en un solo prompt. El agent se va a perder. Mejor: modular la spec, darle secciones relevantes por task.

Red Flag 3: Asumir que si los tests pasan, el código es correcto. Los tests pueden estar incompletos. El código puede pasar tests pero no resolver el problema real. Siempre revisá.

Red Flag 4: Skipear la review del plan. "Eh, el agent sabe lo que hace." No. Revisá el plan siempre. Catching un error en el plan es 100 veces más barato que fixing después.

Red Flag 5: Usar el mismo temperature para todo. Critique y ideación necesitan temperature alta (creatividad). Implementation necesita temperature baja (determinismo). Ajustá según fase.

Red Flag 6: No versionar la spec. Escribís una spec, la cambiás 10 veces, y después no recordás por qué hiciste cierto cambio. Usá Git. Cada decision = un commit con mensaje claro.

Eviten estos red flags y van a tener muchísimo mejor experiencia con coding agents.

Slide 19

[2 minutos]

Ok, cerremos con Quick Wins - cosas que pueden empezar a hacer HOY que van a tener impacto inmediato.

Quick Win 1: Respondé las 3 preguntas ANTES de prompt. ¿Qué quiero? ¿Qué NO quiero? ¿Cómo sé que funciona? Si no podés responder claras, no estás listo para el agent.

Quick Win 2: Hacé al menos 1 round de critique. Aunque sea uno. Prompt simple: "Revisá esta idea. ¿Qué estoy omitiendo?" Ya con eso van a encontrar huecos que no vieron.

Quick Win 3: Escribí verification criteria ANTES de codear. Una tabla simple: requirement → test → success criteria. Esto conecta spec con testing.

Quick Win 4: Usá un quality gate checklist. Score de 25+/30 puntos antes de pasar a planning. Asegurá que la spec está completa.

Quick Win 5: En Claude Code, usá /plan antes de escribir código. Siempre. Review el plan. Si algo no tiene sentido, arreglalo ANTES del código.

Quick Win 6: Escribí tus decisions en un archivo .md. Aunque sean bullets simples. "Decidimos usar Firebase porque..." "No vamos a soportar múltiples monedas porque..." Esto les da memoria a través de sesiones.

Quick Win 7: Probá role-based critique. Una sola vez. "Revisá esto como security engineer." Van a ver qué tipo de issues encuentra que no encontraste vos.

Solo con estos 7 quick wins, sin hacer el proceso completo, ya van a ver mejora inmediata en la calidad de lo que producen con agents.

Y después, a medida que se sienten cómodos, agregan más: multi-round critique, chain of thought, boundaries, context building.

Slide 20

[2 minutos]

Recap rápido de lo que vimos:

De Scope a Spec: Las 3 preguntas fundamentales, fase de ideación colaborando con el agent, bajar a spec con los 6 elementos.

Crítica Multi-Round: Buscar gaps, edge cases, blind spots. Iterar hasta estar satisfechos.

Técnicas: Role-based critique para diferentes perspectivas, Chain of Thought para reasoning profundo, Three-Tier Boundaries para reglas claras.

Model Settings: Temperature alta para critique y exploración, baja para implementation.

Context Building: Constitution del proyecto, strategy docs, style guides. Mantener alignment a través de sesiones.

Y el mensaje final, el más importante:

> "Invertí tiempo en PENSAR antes de EJECUTAR.
>
> La spec es tu pensamiento hecho explícito.
> El agent ejecuta tu pensamiento.
>
> Pensamiento claro = Código claro."

Los coding agents son increíblemente poderosos. Pero su poder depende 100% de la claridad de tu pensamiento.

Si tu pensamiento es confuso, el código va a ser confuso. Si tu pensamiento es claro, el código va a ser claro.

Spec-Driven Development es simplemente la disciplina de hacer tu pensamiento explícito ANTES de ejecutar.

Y con esa disciplina, van a ver que los agents pasan de ser "frustrantes" a ser multiplicadores reales de productividad.
