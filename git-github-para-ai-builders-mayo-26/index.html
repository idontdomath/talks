"Revisá mis cambios antes del merge"

   GitHub Desktop → Create PR    # visual
   "Abrime un PR" al agente      # 50/50

Como pedir feedback en Google Docs, pero con botón "Aprobar".

¿Tengo que usarlos siempre? No. Para empezar, commits directos a main están bien para la mayoría de los cambios. Branches y PRs son herramientas opcionales que tienen cuando las necesitan.
Branches & PRs 11 / 16
Branches cortas · PRs rápidos · main siempre integrada

Trunk-based development

Cambios chicos van a main a través de un PR. La clave: branches que duran horas, no semanas. Nada de mega-merges al final.

Lun 10am  María  $ git checkout -b api-pagos     # branch corta
Lun 11am  María  $ git commit -m "API de pagos"
Lun 12pm  María  $ gh pr create  →  review  →  merge a main ✓

Lun 2pm   Juan   $ git pull                      # trae cambios de María
Lun 2pm   Juan   $ git checkout -b tech-stack
Lun 4pm   Juan   $ gh pr create  →  review  →  merge a main ✓

Cada cambio: branch corta → commit → PR → merge. Repetir muchas veces al día.
✅ Trunk-based · branches de horas, PRs chicos, merges frecuentes. Main siempre verde y al día.
❌ Anti-patrón · branches de semanas, mega-PR de 50 archivos, merges dolorosos con conflicts.

Levanten la mano los que tienen archivos en su computadora con nombres como estos. [pausa para manos]

Git resuelve esto. Es la herramienta que la industria entera usa para no caer en este caos. Y hoy vamos a ver cómo usarla — sin que les duela.

Slide 03

[4 minutos]

Antes que nada, una aclaración que les va a ahorrar mucha confusión: Git no es lo mismo que GitHub. La gente los usa como sinónimos pero son cosas distintas.

Git es un sistema de control de versiones. Es un programa que corre en su computadora. Si ustedes usaron alguna vez "Track Changes" en Word, eso es la idea — pero con esteroides. Git registra cada cambio que hacen y les permite volver atrás cuando quieran.

GitHub es un servicio en la nube — propiedad de Microsoft — que aloja repositorios de Git. Piensen en GitHub como Google Drive, pero específicamente para proyectos de Git. Ahí guardan sus archivos, ahí colaboran con su equipo.

Pueden usar Git sin GitHub (totalmente local), pero para colaborar y para que su agente trabaje bien, necesitan ambos. Ok, sigamos.

[transición al diagrama Netflix]

La mejor analogía que conozco para Git es Netflix.

Imaginen que su proyecto es una serie de Netflix. Cada cambio importante que hacen es un episodio. El lunes agregaron login — episodio 1. El martes arreglaron un bug — episodio 2. El miércoles agregaron un dashboard — episodio 3. Hoy, jueves, están agregando una API de pagos — están grabando el episodio 4.

Cada uno de esos "episodios" en Git se llama un commit. Es una foto del proyecto en ese momento, con una descripción de qué cambió.

Hay una diferencia clave con Netflix, eh: en Netflix los episodios ya existen, ustedes solo los miran. En Git, ustedes son el director. Ustedes deciden cuándo grabar cada episodio. Si trabajaron 3 horas y no commitearon, no hay episodio. Es importante que entiendan esto: Git no graba automáticamente, ustedes decretan cuándo cerrar un episodio.

¿Qué pueden hacer con esto? Tres cosas que les van a salvar la vida:

Primero, pueden ver cualquier episodio anterior. ¿Cómo se veía el proyecto el martes a las 3pm? Lo abren, lo ven.

Segundo, pueden volver atrás. "Esto que agregué hoy rompió todo, volvamos a ayer." Un comando, listo.

Tercero, y esto es lo más útil en el día a día: pueden ver exactamente qué cambió entre dos episodios. "El viernes algo se rompió, ¿qué cambió desde el jueves?" Git les muestra línea por línea.

Esto es Git. Time machine para su proyecto. Sigamos con los conceptos básicos.

Slide 04

[4 minutos]

Hay dos conceptos que si entienden bien, el resto de la charla cae solo. Repository y commit. Vamos uno por uno.

Repository, o "repo". Un repo es básicamente la carpeta de su proyecto. Si tienen un proyecto en `mi-proyecto/`, esa carpeta entera es el repo.

¿Cuál es la diferencia con una carpeta común de Google Drive? Google Drive guarda solo la versión actual de cada archivo. Si editan un archivo y lo guardan, perdieron lo que había antes — bueno, técnicamente Google guarda historial, pero por archivo, no por proyecto.

Un repo de Git guarda los archivos más todo el historial de cambios, del proyecto entero. Es como Google Docs con "ver historial de versiones", pero aplicado a la carpeta completa, no a un solo archivo. Eso les permite ver, en cualquier momento, cómo se veía el proyecto entero hace una semana, hace un mes, hace seis meses.

[pasamos al segundo concepto]

Commit. Esto es el corazón de Git, así que prestemos atención.

Un commit es un cambio guardado con descripción. Es exactamente lo que hablamos en el slide anterior: el "episodio" que ustedes graban.

La analogía que me gusta más para esto es: imaginen que cada commit es una foto de su proyecto en ese momento, y ustedes escriben en el reverso de la foto qué cambió. "Foto 1: agregué login. Foto 2: arreglé un bug. Foto 3: nueva API."

¿Por qué es importante la descripción? Porque en dos semanas, cuando alguien — o ustedes mismos — pregunten "¿por qué cambiamos esto?", abren la historia, leen la descripción, y se acuerdan. Sin descripciones, esto es imposible.

Una regla práctica que les va a salvar la vida: descripción clara, siempre. Más tarde vamos a ver ejemplos de buenas y malas descripciones.

Estos dos conceptos — repo y commit — son el 80% de Git. Si los entienden, ya están. Lo demás es vocabulario.

Slide 05

[1 minuto]

Antes de seguir, vocabulario rápido. No necesitan memorizar nada — solo reconocer cuando los vean en lo que viene.

Clone: copiar un repo desde GitHub a su computadora. Es como descargar una carpeta entera de Drive.

Pull: traerse los cambios nuevos que hicieron otros. Si María hizo cambios y los subió, ustedes hacen pull para tenerlos. Es el "refresh" de Gmail.

Push: lo opuesto. Subir sus cambios a GitHub para que el equipo los vea. Es como guardar en la nube.

Merge: unir cambios de dos lados. Lo van a ver más adelante con branches.

Conflict: cuando dos personas editaron exactamente la misma línea al mismo tiempo. Pasa rara vez, suena peor de lo que es, lo vamos a ver al final.

Listo, ya tienen el vocabulario. Ahora veamos cómo se usa todo esto en el día a día.

Slide 06

[5 minutos]

Acá está la parte más importante de toda la charla. Si se llevan algo de hoy, que sean estos cuatro pasos. Pull, Edit, Commit, Push. Los van a hacer todos los días, miles de veces. Es el 80% del uso de Git.

Vamos uno por uno. Y para cada uno les voy a mostrar dos formas de hacerlo: terminal — escribiendo comandos — y GitHub Desktop — clickeando en una app. Elijan la que les acomode más, hacen exactamente lo mismo.

Paso 1: Pull. Traer los últimos cambios.

Antes de tocar cualquier archivo, traen lo último. Por qué: porque tal vez María subió cambios mientras ustedes dormían, y si editan sin hacer pull, después se les arma un quilombo de conflicts.

En terminal: `git pull`. Una palabra. En GitHub Desktop: click en "Fetch origin" arriba, y si hay cambios, "Pull origin". La analogía es hacer refresh en Gmail antes de leer mails — quieren ver lo último, no lo de hace tres horas.

Paso 2: Edit. Editar archivos.

Esto es lo que ya saben hacer. Abren un archivo en su editor — VS Code, Sublime, lo que sea — y lo modifican. Git no se entera de nada todavía. Editan tranquilos.

Paso 3: Commit. Guardar sus cambios con descripción.

Esto es el "graban el episodio". En terminal son dos comandos. Primero `git add` con el archivo — esto le dice a Git "este archivo lo quiero incluir en el commit". Es como armar el paquete antes de enviarlo. Después `git commit -m` con la descripción entre comillas. La descripción es lo que aparece en la "foto" — pongan algo claro, no "fix" ni "asd".

En GitHub Desktop es todo en una pantalla: ven los archivos cambiados, ponen un check en los que quieren incluir, escriben el resumen, y click en "Commit to main". Listo.

Paso 4: Push. Subir a GitHub.

Hasta aquí todo lo que hicieron está solo en su computadora. Para que el equipo lo vea, hay que subirlo. En terminal: `git push`. En GitHub Desktop: click en "Push origin". Y ya está, sus cambios viven en la nube.

[gesto al diagrama de los 4 pasos]

Pull, edit, commit, push. Cuatro pasos. Es literalmente lo que van a hacer todos los días. No necesitan saber nada más para arrancar. En serio.

Si los comandos de terminal los intimidan, usen GitHub Desktop sin culpa. No es "menos profesional", no es "para principiantes" — hace exactamente lo mismo. Mucha gente que lleva años trabajando con Git usa GitHub Desktop porque es más rápido visualmente.

Esto es lo que hacen ustedes. Ahora veamos qué hace su agente.

Slide 07

[2 minutos]

Llegamos a la mitad de la charla. Hasta aquí vimos el lado humano. Ahora veamos el lado del agente. Este bloque vale el mismo tiempo que el anterior porque el agente es la mitad del valor.

Lo primero, antes que nada, es setup. Su agente — Claude Code, Cursor, Codex, el que sea — necesita dos herramientas instaladas en su computadora para poder trabajar con Git: Git y GitHub CLI, que es un comando que se llama `gh`.

¿Por qué? Porque su agente, cuando hace cambios al proyecto, necesita poder hacer commits, crear branches, abrir Pull Requests. Y la única forma que tiene de hacer eso es ejecutando los mismos comandos que ustedes podrían escribir. Si Git no está instalado, el agente literalmente no puede commitear nada. Va a hacer cambios en archivos, pero esos cambios se quedan en el limbo, sin guardarse en el historial.

Las instalaciones son rápidas. En Mac es `brew install git` y `brew install gh`. En Windows, `winget install`. En Linux, `apt-get install`. Tienen los comandos exactos en el slide.

Y después una vez: `gh auth login`. Esto autentica el comando `gh` con su cuenta de GitHub. Sigan las instrucciones en pantalla, copian un código, lo pegan en el browser, listo.

Esto es una sola vez en la vida. Después se olvidan, y su agente lo usa solo cuando lo necesita.

Una aclaración importante: GitHub Desktop no aplica para los agentes. Los agentes no clickean botones en una app — ellos ejecutan comandos. Así que aunque ustedes prefieran GitHub Desktop para su trabajo, igual tienen que instalar Git y `gh` para que su agente funcione. Es requisito.

Listo, ya están seteados. Ahora veamos qué hace el agente con esto.

Slide 08

[3 minutos]

Acá viene la parte que más confusión genera con los agentes: ¿qué hace el agente solo, sin que yo lo toque?

La respuesta corta: lo mismo que hacen ustedes en el slide anterior. Pull, edit, commit, push. Pero usándolo por ustedes.

La forma más útil de pensar al agente es esta: es un junior eficiente que tiene acceso a las mismas herramientas que ustedes. Lee el repo, edita archivos, hace commits con mensajes que él mismo escribe, crea branches, hace push, y si le piden, abre Pull Requests.

Veamos un ejemplo concreto. Imaginen que le dicen al agente: *"Agregá la API de pagos al AGENTS.md"*. Una sola frase. Lo que hace el agente, paso por paso:

Primero, lee el AGENTS.md actual para entender qué hay. Después edita el archivo, le agrega la sección de pagos. Después corre `git add AGENTS.md` para preparar el commit. Después `git commit -m "Add payments API to AGENTS.md"` — fíjense que el mensaje lo escribió él, no ustedes. Después `git push` para subirlo. Y al final les dice: *"Listo, hice commit y push. Acá el diff por si quieren revisar"*.

¿Cuánto tardó esto? Veinte segundos. ¿Cuánto les hubiera tardado a ustedes? Cinco minutos.

Acá está la idea importante que quiero que se lleven: el agente no es magia. El agente está corriendo, literalmente, los mismos comandos que vimos en el slide anterior. `git add`, `git commit`, `git push`. No hay un "modo especial AI". Por eso necesita Git y `gh` instalados — porque sin ellos no tiene cómo ejecutar nada.

Esto tiene una consecuencia importante: si entienden Git mínimamente — los cuatro pasos del slide anterior — entienden lo que está haciendo su agente. Pueden mirar lo que hizo y validar. No es una caja negra.

Pero ojo: el agente puede equivocarse. Puede commitear algo que no era. Puede pisar trabajo de otra persona. Puede pushear sin que ustedes hayan revisado. Por eso el siguiente slide es crítico: ustedes son el code reviewer.

Slide 09

[2 minutos]

OK, el agente puede hacer mucho solo. La pregunta natural — y muy válida — es: ¿no pierdo el control?

La respuesta corta: no, ustedes mandan en cada paso. Tienen tres mecanismos de control, y los voy a explicar uno por uno.

Primero: aprobación por herramienta. Cuando el agente quiere ejecutar algo "serio" — un commit, un push, abrir un PR — les pregunta antes. Aparece una notificación en su Claude Code, en su Cursor, lo que sea, que dice: *"Voy a correr `git commit -m 'X'`. ¿OK?"* Y ustedes deciden: aprobar este caso, aprobar siempre — eso se llama "auto-approve" — o rechazar y darle feedback. Para acciones que pisan main directamente, mi recomendación es no activar auto-approve al principio. Aprueben caso por caso hasta que confíen.

Segundo: revisar el diff. Antes de pushear, el agente les muestra qué cambió. Líneas en verde, lo que agregó. Líneas en rojo, lo que sacó. Lo leen, lo entienden, lo validan. Si algo no les cierra, le dicen *"esto no, hacelo así"* y el agente corrige. No tienen que ser developers para leer un diff — verde es agregado, rojo es sacado. Punto.

Tercero: para cambios grandes, Pull Requests. En vez de pushear directo a main, le piden al agente que abra un PR. Eso es un mecanismo de revisión más formal — vamos a verlo en el próximo slide. Útil cuando el cambio es grande o cuando otra persona del equipo necesita ver qué hicieron.

Y aquí está el mental model que quiero que se lleven: el agente es un junior que ejecuta. Ustedes son el senior que aprueba el merge. El agente puede teclear cien veces más rápido que ustedes, pero el criterio — qué entra y qué no — es de ustedes.

Si activan auto-approve para todo desde el día uno, están dándole al junior las llaves del auto sin license. Mal idea. Empiecen aprobando caso por caso. Con el tiempo van a saber en qué confía y en qué no, y ahí van ajustando.

Listo, eso es la sección del agente. Volvamos al lado más práctico: ¿cuándo conviene usar branches y PRs?

Slide 10

[4 minutos]

Hasta aquí vimos commits directos a main. Esto funciona muy bien para cambios chicos, fixes obvios, cosas en las que confían. Pero hay momentos donde no quieren commitear directo. Por ejemplo: están probando algo que puede no funcionar. O el cambio es grande y quieren que alguien lo revise antes de que entre. O van a tardar varios días en terminarlo y no quieren tener main "rota" mientras tanto.

Para esos casos existen dos cosas: branches y Pull Requests. Vamos una por una.

Branch. Una branch es una línea de tiempo paralela del proyecto. Piénsenlo como un borrador antes del email final. La línea principal — main — sigue siendo la versión "oficial" del proyecto. Ustedes crean una branch nueva, le ponen un nombre — por ejemplo `experimento-pagos` —, y trabajan ahí. Hacen los commits que quieran, pueden ser cinco, pueden ser veinte, no importa. Main no se entera de nada de lo que están haciendo.

[gesto al diagrama]

Acá lo ven: main sigue avanzando con sus commits "oficiales", y por debajo va la branch del experimento, separada. Cuando deciden si funcionó o no, pasan al segundo concepto.

Pull Request, o PR. Un PR es básicamente decir: *"Equipo, terminé mi trabajo en esta branch, por favor revísenlo antes de que lo agregue a main"*. Es un mecanismo de revisión formal.

El flujo es: trabajan en su branch, cuando están listos abren un PR — o le piden al agente que lo abra. El PR aparece en GitHub, alguien del equipo lo ve, mira el diff, deja comentarios, y al final aprueba o pide cambios. Cuando está aprobado, hacen click en "Merge" y los cambios entran a main.

¿Cómo abren un PR? Tres formas. En terminal: `gh pr create`. En GitHub Desktop: hay un botón "Create Pull Request". O le dicen al agente: *"Abrime un PR"*, y lo abre solo.

La analogía que mejor funciona: es como pedir feedback en un Google Doc, pero más formal y con un botón "Aprobar" explícito.

Una pregunta natural: ¿siempre tengo que usar branches y PRs? No. Para AI Builders, al principio, commits directos a main están perfecto para la mayoría de los cambios. Branches y PRs son herramientas que tienen disponibles cuando las necesitan — para experimentos riesgosos, cambios grandes, o trabajo que requiere revisión.

Veamos esto un poco más en el próximo slide.

Slide 11

[1 minuto]

Esto que les acabo de describir — commits chicos, branches cortas, PRs frecuentes — tiene un nombre en la industria: trunk-based development. Es el patrón que usan equipos como Google, Facebook, y la mayoría de teams modernos.

La idea clave es esta: los cambios entran a main a través de un PR, pero las branches duran horas, no semanas. Hacen un cambio chico, abren branch, commit, abren PR, alguien lo aprueba en cinco minutos, merge. Y al rato, otro cambio. Y otro. Diez veces al día si hace falta.

¿Qué se evita con esto? El anti-patrón clásico: branches que viven dos semanas, mega-PRs de cincuenta archivos, y un merge gigante al final que rompe la mitad de las cosas y nadie sabe por qué.

Las ventajas son concretas: menos conflicts porque los cambios son chicos, main siempre está al día, todos ven los cambios de los demás rápido, y nunca hay merges dolorosos.

Una aclaración para que arranquen sin fricción: están en un entorno de aprendizaje. En los repos donde van a practicar, los PRs van a ser informales — pueden auto-aprobarse, no van a tener todas las protecciones formales que tiene producción. La idea es que se acostumbren al flujo: branch → commit → PR → merge. Las protecciones reales — branch protection rules, approvals obligatorios, status checks — van a ir apareciendo después.

Sigamos con algo que probablemente les preocupa: ¿qué pasa cuando hay conflicts?

Slide 12

[2 minutos]

Conflicts. La palabra suena fea. Mucha gente le tiene pánico a esto. Pero les voy a contar la verdad: los conflicts son raros y se resuelven en treinta segundos.

¿Qué es un conflict? Cuando dos personas — o un humano y un agente — editaron exactamente la misma línea del mismo archivo al mismo tiempo. La analogía que mejor funciona: dos personas hablando al mismo tiempo en una llamada de Zoom. Git no puede adivinar a cuál escuchar, así que les dice: *"Ey, ambos editaron esto. ¿Cuál versión quieren?"*

[mostrar el ejemplo del slide]

Cuando esto pasa, Git les marca el archivo con esos símbolos raros — los `<<<<`, `====`, `>>>>`. Al principio asusta, pero es muy simple: arriba está la versión de uno, abajo está la versión del otro, y ustedes deciden cuál queda — o si combinan ambas.

Cómo lo resuelven, dos formas:

Forma fácil: GitHub Desktop. Detecta el conflict automáticamente, les dice "abran esto en VS Code", y en VS Code aparecen botones grandes: "Accept Current" — quédense con la mía —, "Accept Incoming" — quédense con la del otro —, "Accept Both" — combinen las dos. Click, listo, commit.

Forma manual: terminal. Abren el archivo, deciden qué versión queda — o combinan —, borran las líneas raras con los símbolos, guardan, y hacen `git add` y `git commit`. También treinta segundos.

Ahora la parte tranquilizadora. Tres cosas:

Primero, Git les avisa antes de pisar el trabajo de alguien. No hay forma de "destruir" sin querer.

Segundo, conflicts pasan rara vez. En la práctica, si cada uno edita su parte del proyecto, conflicts son uno por mes — si eso. La gente que conoce Git bien casi nunca los ve.

Tercero, siempre pueden pedir ayuda. Si ven un conflict y se atascan, pegan el screenshot en `#ai-builders` Slack y alguien los ayuda. Nadie va a juzgarlos por preguntar.

Sigamos. Ahora un tour rápido por GitHub.com para que sepan qué ven cuando entran.

Slide 13

[2 minutos]

Rápido tour por GitHub.com. Probablemente ya entraron a GitHub alguna vez, pero quizás no entendieron bien qué era cada cosa. Este es el mapa.

[señalar al diagrama del repo]

Cuando abren un repo en GitHub, lo que ven es esto: arriba el nombre del proyecto, una fila de tabs, y abajo la lista de archivos.

Tres tabs van a usar todo el tiempo:

Code. El default. Acá están los archivos del proyecto, y abajo de cada archivo dice cuál fue el último commit que lo tocó y quién. Útil para responder "¿qué se cambió últimamente?".

Pull Requests. Acá viven los PRs abiertos y los cerrados. Si alguien abrió un PR para que ustedes lo revisen, lo ven aquí. También pueden ver los PRs viejos para entender el historial de cambios grandes.

Actions. Acá viven las automatizaciones — cuando ustedes pushean, GitHub corre tests, validaciones, deploys, todo eso aparece en Actions. Útil para chequear si algo falló después de un push.

[pasar al diagrama del PR]

Ahora la pantalla de un Pull Request, que es la que más les va a importar.

Arriba el título y quién lo abrió. Después la lista de archivos cambiados. Y abajo el diff — la parte importante. Verde es lo que se agregó, rojo es lo que se sacó. Lo leen, lo entienden, y al final tienen tres botones: Approve para aprobar, Request Changes para pedir cambios, o Comment para dejar un comentario sin bloquear.

Cuando un PR está aprobado, el botón Merge se habilita y los cambios entran a main.

Esto es básicamente todo lo que necesitan saber del web. Lo demás se aprende clickeando. Avancemos a las reglas que les van a evitar el 80% del sufrimiento.

Slide 14

[2 minutos]

Cinco reglas que si las siguen, evitan el 80% de los problemas que tiene la gente con Git. Las repaso rápido.

Uno: pull antes de editar. Siempre. Es lo primero que hacen al sentarse a trabajar. `git pull`. Esto evita conflicts antes de que existan.

Dos: commits chicos y frecuentes. No esperen dos semanas para hacer un mega commit con cincuenta cambios. Vayan commiteando cada vez que terminan algo concreto. "Agregué API de pagos", commit. "Actualicé el tech stack", commit. "Documenté el proceso de testing", commit. Tres commits, tres pasos claros. Versus un único commit "cambios varios" que en tres meses no entiende nadie.

Tres: mensajes de commit claros. Esto es la regla que más se viola. La pregunta que tienen que hacerse es: *"En tres meses, ¿voy a entender qué hice aquí?"*. Si el mensaje es "fix" o "asd" o "updates", la respuesta es no. Pongan algo descriptivo aunque les tome 5 segundos extra.

Cuatro: no editen archivos globales sin avisar. Si están en su propio AGENTS.md, en su propio README, editan libremente. Pero si tocan archivos compartidos del equipo o globales de la empresa, abran un PR o pregunten primero. Esto es cortesía y evita pisar trabajo de otros.

Cinco: si no están seguros, branch. Cualquier cosa que les genere duda — cambio grande, experimento, "no sé si va a funcionar" — eso va en branch + PR. El costo de abrir un branch es cero, el costo de romper main es alto.

[transición a errores comunes]

Y ya que estamos, los cuatro errores más comunes que van a ver, y cómo arreglarlos.

`fatal: not a git repository`. Significa que están en la carpeta equivocada. `cd` al proyecto correcto y listo.

`Your branch is behind 'origin/main'`. Alguien más hizo cambios que ustedes no tienen. `git pull` para traerlos.

`Permission denied (publickey)`. GitHub no los reconoce. Esto requiere configurar credenciales — pidan ayuda en `#ai-builders`, los ayudamos en cinco minutos.

`Changes not staged for commit`. Editaron archivos pero no hicieron `git add`. Solución: `git add` con el archivo, después `git commit`.

Regla maestra: si ven un error que no entienden, copia, pega en `#ai-builders` Slack. Alguien ya lo vio antes, alguien sabe la solución.

Slide 15

[1 minuto]

Recap rápido de qué cubrimos.

Git es time machine para su proyecto: commits son fotos, GitHub es donde viven esas fotos en la nube.

El workflow básico de ustedes son cuatro pasos: pull, edit, commit, push. Eso es el 80% de su uso de Git en el día a día.

Su agente — Claude Code, Cursor, Codex — usa exactamente los mismos comandos que ustedes. No es magia. Por eso necesita git y `gh` instalados. Y ustedes son el code reviewer: aprueban los commits, leen los diffs, deciden qué entra.

Branches y Pull Requests son herramientas que tienen disponibles cuando las necesitan: experimentos riesgosos, cambios grandes, cosas que requieren revisión. No son obligatorias para todo.

Conflicts pasan poco y se resuelven en treinta segundos.

Y cinco reglas para no sufrir: pull antes de editar, commits chicos y frecuentes, mensajes claros, no editar archivos globales sin avisar, branch para experimentos.

Eso es todo. Vamos a los próximos pasos.

Esta semana, cuatro cosas concretas. Instalen Git y `gh` CLI — el slide 6 tiene los comandos exactos para Mac, Windows y Linux. Cloneen su primer repo — el de práctica que vamos a compartir hoy. Practiquen el workflow básico: pull, edit, commit, push. Y configuren Git en su agente, sea Claude Code, Cursor o el que usen.

## Q&A — Preguntas Frecuentes Anticipadas

(Para responder en Q&A si surgen — no son slides)

P: ¿Qué pasa si rompo algo?
R: Git guarda TODO. Siempre pueden volver atrás. No hay forma de "romper permanentemente".

P: ¿Puedo trabajar offline?
R: Sí. Git funciona local. Hacés commits offline, push cuando tengas internet.

P: ¿Necesito memorizar comandos?
R: No. Usá GitHub Desktop, o copia/pega del cheat sheet (handout). Con práctica se vuelve automático.

P: ¿Cuándo usar branch vs main directo?
R: Main directo para cambios chicos seguros. Branch para experimentos o cambios que pueden romper algo.

P: ¿Qué pasa si commiteo un .env con secrets?
R: GitHub bloquea el push automáticamente. Además `.gitignore` previene esto. Si pasara, Platform Team lo limpia. No entres en pánico.

P: ¿GitHub Desktop o terminal?
R: Para ustedes: cualquiera, GitHub Desktop es más fácil al inicio. Para tu agente: necesita git + gh CLI sí o sí (no usa GitHub Desktop).

P: ¿Mi agente puede romper mi repo?
R: No, ustedes aprueban cada commit/push. Y aunque pasara, todo es reversible (Git es time machine).

P: ¿Puedo recuperar algo que borré hace 2 meses?
R: Sí. Si fue commiteado, Git lo tiene. Pedí ayuda en #ai-builders y lo recuperamos.

P: ¿Puedo ver quién escribió cada línea de código?
R: Sí — se llama "Git Blame". En GitHub: archivo → "Blame" → ves autor de cada línea.
