Slide 01

[1 minuto]

Hola a todos. La charla anterior fue sobre Git/GitHub — la base que necesitan para colaborar en código. Hoy vamos al siguiente nivel: el harness del programa AI Builders, que es el primer entorno real donde lo van a usar.

En 30 minutos vamos a ver tres cosas. Primero: qué es el harness y cómo lo van a usar en sus proyectos. Segundo: cómo se le va a agregar contexto de trabajo a medida que el programa crece. Tercero: qué herramientas y guardrails tiene su coding agent cuando trabaja sobre los proyectos del programa.

La idea es que al final del bloque tengan claro por dónde arrancar su primer proyecto, cómo extender el template con contexto específico, y cómo el harness les ahorra el setup repetitivo.

Slide 02

[3 minutos]

Para entender qué es un harness, piensen en un template de Google Docs. Cuando arrancan un documento desde un template — una propuesta comercial, un kickoff de proyecto — no parten de una hoja en blanco. El template ya trae el logo de la empresa, las secciones con headers, el formato. Algunas partes son fijas (logo, footer legal) y otras les piden llenar contenido específico ("aquí va el nombre del cliente"). Si modifican las fijas manualmente, el próximo update del template las sobreescribe.

El harness del programa AI Builders es exactamente eso, pero para proyectos enteros de código. Cuando arrancan un proyecto del programa, no parten de un repo vacío. Reciben un repo donde ya están:

- La estructura de carpetas armada.
- El contexto de Habi cargado para que el agente sepa quién es Habi, qué productos tenemos, cómo trabajamos.
- La configuración de seguridad lista.
- Secciones marcadas donde ustedes agregan el contexto específico del proyecto: "este proyecto usa Stripe", "este proyecto tiene un workflow de aprobación", etc.

Cuando Plataforma actualiza el template, ustedes reciben las mejoras sin perder su contexto de proyecto. Las dos capas conviven: la program-level que evoluciona, y la project-level que es de ustedes.

Slide 03

[2 minutos]

El harness es la diferencia entre construir cada proyecto desde cero y construir sobre una base compartida.

Sin harness, cada vez que arrancan un proyecto, su agente parte sin saber nada de Habi. Tienen que explicarle qué productos tenemos, cuáles son nuestras conventions, cómo manejamos seguridad. Ustedes terminan repitiendo el setup en cada proyecto. Y si Plataforma mejora algo, tienen que ir proyecto por proyecto aplicándolo.

Con el harness, todo eso ya está cargado el primer día. El agente abre el proyecto y ya sabe qué es Habi. Las prácticas de seguridad están activas y no dependen de que ustedes las recuerden. La estructura es la misma entre proyectos. Y cuando Plataforma actualiza algo, corren un comando y reciben las mejoras.

El punto importante: el harness les saca de encima el trabajo de armar cada proyecto desde cero. Ustedes se concentran en el problema del proyecto, no en el plumbing.

Slide 04

[3 minutos]

Antes de entrar en el flujo, una regla del programa: todo proyecto que armen como AI Builder vive en un repositorio de la organización HabiGlobal, incluso si es solo un prototipo mínimo o una prueba de concepto. No trabajamos en repos personales o en local sin push — todo lo que se construye dentro del programa queda en `HabiGlobal/`. Esto nos da visibilidad, governance compartida y la capacidad de aplicar mejoras del template a todos los proyectos a la vez.

Con esa premisa clara, el flujo es así.

Primero, le piden a Plataforma que les arme el proyecto. La información mínima a compartir: nombre del proyecto, descripción corta de qué hace, y qué coding agent oficial van a usar.

Segundo, Plataforma crea el repositorio en la organización HabiGlobal con el naming convention del programa: `internal-<nombre>`. Por ejemplo, `internal-property-search`, `internal-lead-router`. Todos los repos del programa llevan ese prefijo `internal-` para identificarlos rápidamente y, eventualmente, poder moverlos a entorno interno cuando estén listos. Plataforma también les configura los permisos como miembros del programa.

Tercero, ustedes reciben el acceso al repo automáticamente. Ese repo ya viene con la estructura del harness completa: las carpetas con el contexto de Habi, los archivos de configuración del agente, los hooks de seguridad — todo listo.

Cuarto, el setup local toma cinco minutos. Clonan el repo, instalan los pre-commit hooks, y ya están listos. Cuando el agente abre el proyecto, carga el contexto de Habi automáticamente.

Si quieren tener más de un proyecto del programa, repiten el flujo. Cada proyecto vive en su propio repo `internal-<nombre>`.

Slide 05

[3 minutos]

La parte central del harness es lo que llamamos Layer 1: el contexto organizacional. Cuando abren el proyecto, el agente tiene acceso a una carpeta `habi-harness/` con cuatro archivos que le dan, en conjunto, todo lo que necesita saber sobre Habi para trabajar con sentido.

El primero es HABI_CONTEXT. Acá está el "quién es Habi": el negocio, los productos que tenemos (Market Maker, Habicredit, Habimetro, Inmobiliaria, Pulppo), la cultura, los principios operativos, la marca, el estilo de escritura. Si le piden al agente que redacte un mail de comunicación al usuario, el agente arranca con el tono correcto porque consultó este archivo.

El segundo es HABI_PLATFORM. Acá está el "cómo se programa en Habi": la arquitectura que usamos, los patterns de testing, deployment, code conventions, security practices. Si le piden al agente que arme un endpoint con FastAPI, arranca con las convenciones que ya tenemos definidas — no inventa.

El tercero es HABI_GUIDELINES. Es un índice curado, pensado específicamente para AI Builders. Funciona como un menú: "para writing → consultar HABI_CONTEXT", "para code → consultar HABI_PLATFORM", "para security → HABI_SECURITY". El agente lo usa para navegar el resto del contexto eficientemente.

El cuarto es HABI_SECURITY. Los guardrails de seguridad: no commitear secrets, no loggear PII, no exfiltrar código fuera de Habi, sandbox de filesystem. Son los cuatro límites que están siempre activos.

Lo importante: estos cuatro archivos los mantiene Plataforma. No tienen que ir a buscar el último canon de Habi — lo reciben en el proyecto.

Slide 06

[3 minutos]

El harness no solo carga el contexto de Habi para el agente — también trae la seguridad automática del programa. Tres capas que funcionan en background sin que ustedes tengan que pensarlas.

Primero, los checks pre-commit. Antes de que un commit se cierre — ya sea uno que ustedes hacen manualmente o uno que el agente hace por ustedes — se ejecutan automáticamente algunos chequeos. Buscan si hay secretos en el código (cosas como API keys, tokens, contraseñas), si hay archivos `.env` con credenciales reales, si hay claves privadas embebidas, archivos enormes, conflictos sin resolver. Si encuentran algo, el commit no entra. Tienen que corregirlo y volver a intentar.

Segundo, las reglas para el coding agent. Cada agente oficial del programa llega configurado de fábrica. Claude Code rehúsa leer archivos sensibles — si le piden "miremos qué hay en el `.env`", responde "no puedo, está bloqueado". Cursor tiene reglas equivalentes. Esto significa que el agente, incluso si ustedes se distraen, no va a abrir un archivo de credenciales y dejarlo en su contexto.

Tercero, archivos sensibles ocultos. En el caso de Gemini, los archivos sensibles son literalmente invisibles para el agente — ni siquiera aparecen en su listado de archivos. Es como si no existieran. Lo que el agente no ve, no puede commitear.

El resultado de las tres capas: ustedes no tienen que recordar "no leer el .env" o "no commitear ese token". La seguridad está activa desde el primer minuto. Si algo se les pasa, se bloquea automáticamente. La protección no depende de su memoria — depende del template.

Slide 07

[3 minutos]

Hasta ahora vimos lo que el harness les da. Ahora vamos a lo que ustedes le agregan al harness.

Cada proyecto del programa tiene tres archivos de guía que el coding agent lee al arrancar: CLAUDE.md, GEMINI.md y AGENTS.md. Los tres tienen la misma estructura interna: dos secciones claramente marcadas con emojis y comentarios.

La primera sección — la que tiene el candado 🔒 — es program-level. Acá están las reglas que aplican a todo proyecto del programa: las invariantes de seguridad, la branch strategy, el comportamiento esperado del agente con Habi. Esta sección la mantiene Plataforma — si la modifican manualmente, el sync lo va a detectar y pedir reconciliación.

La segunda sección — la que tiene el lápiz ✏️ — es project extensions. Esto es su espacio. Es donde agregan todo lo que el agente necesita saber para trabajar bien en _este_ proyecto puntual. Por ejemplo: "este proyecto usa Stripe para pagos, los tests viven en `tests/` y corren con pytest, hay un workflow de approval antes de mergear a main". Cualquier convención, librería, regla específica del proyecto va aquí. Esta sección no se toca cuando Plataforma actualiza la parte program-level — las dos capas conviven.

Son tres archivos distintos porque cada coding agent oficial lee uno principalmente (Claude → CLAUDE.md, Gemini → GEMINI.md, AGENTS.md → varios). En la práctica, pueden poner el mismo contenido en los tres si usan más de un agente, o concentrarse en el archivo del agente que más usan.

En resumen: el harness les da una base común; ustedes le agregan el contexto específico del proyecto en la sección marcada para eso.

Slide 08

[2 minutos]

El último mecanismo es el sync. Plataforma sigue evolucionando el template todo el tiempo — agregando nuevas guías de Habi, mejorando la seguridad, corrigiendo cosas que se detectan en otros proyectos del programa. Ustedes reciben esas mejoras corriendo un solo comando: `bash scripts/sync-harness.sh`.

Cuando corren el sync, dos cosas pasan. Primero, los archivos que mantiene Plataforma — los configs de seguridad, los archivos de Layer 1, los hooks — se reemplazan con la versión nueva del template. No hay nada que decidir ahí; el sync los actualiza automáticamente.

Segundo, para los archivos de guía donde ustedes tienen su sección project-level — CLAUDE.md, GEMINI.md, AGENTS.md — el sync compara con la nueva versión del template. Si detecta cambios, les muestra el diff y les pide que reconcilien manualmente: "estos son los cambios del template, estos son los suyos, decidan qué incorporar". Es un trade-off: más control, menos automatización, porque ahí ustedes tienen contenido propio que no se puede sobreescribir a ciegas.

La analogía: el sync es como una update del sistema operativo. El sistema mejora, pero sus aplicaciones y archivos personales se mantienen. El proyecto se queda fresco sin que tengan que rehacerlo desde cero cada vez que Plataforma libera una mejora.

Slide 09

[2 minutos]

Una regla del programa que vale la pena enfatizar: trunk-based development con GitHub Flow — lo mismo que vieron en la charla anterior, pero acá no es opción, es regla del programa. Una branch principal, feature branches cortas (idealmente menos de una semana), PR a main, merge, delete.

Lo que no usamos: gitflow, branches `develop`/`release`/`hotfix` permanentes, feature branches que viven semanas o meses. Si una branch dura más de una semana, es señal de que hay que cortar el trabajo en pedazos más chicos.

El agente conoce esta convención — está documentada en `HABI_GUIDELINES.md`. Si le piden que cree una branch `develop`, puede recordarles que el programa usa trunk-based.

Un detalle operacional para esta primera cohorte: la regla opera como convención, no hay enforcement automático de branch protection sobre `main`. A partir de la próxima cohorte el enforcement se activa: PR + 1 approval + checks antes de mergear. Lo que practiquen esta semana, después va a estar enforced.

Slide 10

[2 minutos]

El programa tiene tres coding agents oficiales: Claude Code, Cursor, y Gemini CLI. Los tres están validados por Cyberseguridad y tienen licencias resueltas para que ustedes los usen.

Claude Code es el agente de Anthropic. Tiene dos formatos: una CLI que se usa desde la terminal, y un plugin para los IDEs principales (VS Code, JetBrains, etc.). Para pedir licencia, me escriben a mí — soy el owner del seat allocation para Claude en el programa.

Cursor es un IDE standalone. Es como un VS Code pero con el agente integrado nativamente. Para pedir seat, también me escriben a mí.

Gemini CLI es el agente de Google. Se usa desde terminal y está conectado a la cuenta corporativa de Google de Habi — si tienen su cuenta `@habi.co` activa, ya tienen acceso. Si hay algún problema de licenciamiento o acceso, escribir a Víctor Pinzón de Cyberseguridad.

La parte importante para ustedes: el harness aplica de la misma manera a los tres. Las reglas de seguridad están configuradas para los tres agentes. El contexto de Habi está disponible para los tres. Si arrancan con Claude y después prueban Cursor, no hay que cambiar nada del proyecto — el shape se mantiene. Pueden elegir según preferencia, según el caso de uso, o probar varios y decidir.

Slide 11

[3 minutos]

El harness que les estamos entregando hoy es versión 1. Va a seguir creciendo a lo largo del programa, en tres dimensiones.

Primero, más contexto de Habi en el Layer 1. Hoy tienen 4 archivos en `habi-harness/`: CONTEXT, PLATFORM, GUIDELINES y SECURITY. A lo largo del programa esos se van a enriquecer: aid-sdlc va a tener más material sobre cómo programamos en Habi, Cyberseguridad va a expandir el contenido de SECURITY con sus políticas concretas, y vamos a sumar archivos nuevos cuando tenga sentido — un catálogo de APIs internas, un puntero al modelo de datos de OCD, lo que el agente necesite para no inventar nada de Habi.

Segundo, más capacidades en la seguridad y tooling. Hoy tienen el baseline mínimo: detección de secrets, deny patterns, branch strategy. A futuro vamos a sumar restricciones más finas según el tipo de proyecto, quality gates antes de mergear, integración con el data sandbox que está armando OCD, y métricas para entender cómo está corriendo el programa. Todo eso llega vía sync.

Tercero, más AI Builders en el programa. El rollout es en cohortes. Ustedes son la primera, esta semana — 4 o 5 personas. Validamos con ustedes el flow, vemos qué fricciones aparecen, qué falta. La próxima cohorte arranca el lunes próximo, entre 10 y 15 personas. Y a partir de junio, el programa se abre a todos los AI Builders de Habi.

El punto importante: ustedes reciben las mejoras vía sync, sin tener que rehacer nada. Y — esto es clave — el feedback de esta cohorte define qué se prioriza después. Las fricciones que detecten ustedes son los inputs para lo que Plataforma desarrolla la próxima semana. Por eso es importante reportar todo: lo que no funciona, lo que falta, lo que sobra.

Para cerrar: los próximos pasos están listados arriba — request del repo a Plataforma, setup local, cargar el canon, empezar a construir, retro al cierre de la semana. La retro es importante: alimenta lo que arreglamos para C2.

Dos puntos importantes. Primero, sobre pasar el proyecto a producción. Segundo, sobre dónde va el programa. Lo que están haciendo hoy es el primer paso del camino pavimentado hacia entorno interno de Habi. Cuando lleguemos a la próxima versión del programa, sus proyectos van a poder graduar a entornos internos — sandbox primero, después interno — sin tener que re-arquitecturar el proyecto. El shape del harness está pensado desde el inicio para esa transición.

Slide 13

[abierto · ~5 minutos según tiempo]

Espacio abierto para preguntas. Algunas que pueden venir y respuestas cortas:

- "¿Y si quiero usar otro agente?" → En C1 nos quedamos con los tres oficiales (Claude, Cursor, Gemini). Si emerge necesidad concreta, lo evaluamos en la retro.
- "¿Qué pasa si rompo el harness sin querer?" → Los archivos mecánicos están protegidos a nivel de tool y a nivel de pre-commit. Es difícil romperlos sin notarlo. Si pasa, el sync los restituye.

- "¿Cuándo viene el próximo sync?" → Vamos a comunicarlo en la Comunidad de AI Builders en Google Chat cuando haya una versión nueva del template. No es push automático — ustedes deciden cuándo correr el sync.
